Why one framework works for both

Problems have a shape, regardless of domain.

Every unsolved problem — a production outage or a hard conversation you're avoiding — shares the same failure mode: reacting to the symptom instead of understanding the system. A server restart "fixes" a memory leak the same way avoiding a friend "fixes" a conflict — temporarily, and at compounding cost.

The six-stage framework below doesn't care whether the system is made of code or of people. Each stage is presented through both lenses side by side, because the discipline of moving through them in order — not the domain — is what actually solves things.

01

Define — name the real problem, not the symptom.

Most problems are solved badly because they're never actually defined. Write the problem down in one sentence before doing anything else.

💻
Technical Lens

Separate the symptom ("the app is slow") from the problem statement ("p95 checkout latency exceeds 2s under load, since deploy #4471"). Include scope, timeline, and impact — not a guess at the cause.

A well-formed technical problem statement is falsifiable: someone else could look at it and know exactly what "solved" looks like.

"Checkout latency p95 > 2000ms since 14:32 UTC, affecting 8% of EU traffic. Was 340ms baseline. Not yet: cause unknown."
💗
Personal Lens

Separate the feeling ("I'm always exhausted") from the problem statement ("I've said yes to 4 commitments this month I didn't want, and I don't know how to say no without guilt").

A well-formed personal problem statement names the specific situation and the specific gap — not a global judgment about yourself.

"I feel resentful after every family call, specifically when asked about my career. I haven't set a boundary about this topic."
Shared principle: If you can't write the problem as one falsifiable sentence, you don't understand it yet — you're still reacting to it.
02

Diagnose — find root cause, resist the first explanation.

The first explanation is almost always the most available one, not the most accurate one. Diagnosis is the discipline of going one layer deeper than feels necessary.

💻
Technical Lens

Use the 5 Whys against logs, metrics, and traces — not intuition. Reproduce before you fix. Bisect the change history (deploys, config, traffic pattern) to isolate the variable that actually correlates with the symptom.

Correlation-first, causation-second: confirm the fix actually addresses the mechanism, not just the timeline.

Why slow? → DB query slow. Why? → Missing index. Why missing? → New query path added in #4471. Why untested? → No load test on that endpoint.
💗
Personal Lens

Use the 5 Whys against your own reaction, gently. The first answer is usually a story you've told yourself before, not the actual cause. Ask what the feeling is protecting you from.

Pattern-first, incident-second: is this a one-off, or the fifth time the same trigger has produced the same reaction?

Why resentful? → Feels like judgment. Why judgment? → Comparison to my sibling. Why sensitive to it? → I already doubt this choice myself.
Shared principle: A fix applied to the wrong cause doesn't just fail — it teaches you the wrong lesson and makes the next diagnosis harder.
03

Diverge — generate options before committing to one.

The second solution you think of is rarely better than the first — until you deliberately force yourself past it. Aim for at least three genuinely different approaches.

💻
Technical Lens

Generate options at different trade-off points: the 10-minute mitigation, the 1-day proper fix, and the 1-week structural fix that prevents the whole class of bug. Write all three down before choosing.

Ask: what would this look like if compute/time were free? What would it look like with zero budget? The extremes reveal the real constraint.

A) Rollback deploy #4471 (5 min). B) Add missing index (2 hrs). C) Add load-testing gate to CI (1 week, prevents recurrence).
💗
Personal Lens

Generate options across a spectrum: the smallest possible step, the ideal outcome if fear weren't a factor, and the compromise that respects both people's needs. Write all three down before choosing.

Ask a trusted person for one option you haven't considered — the options you can't see are usually the ones your own bias is hiding.

A) Say nothing, redirect the topic (avoids conflict). B) State the boundary directly next call. C) Write it in a message first, so it's said calmly.
Shared principle: If you only generated one option, you didn't diverge — you rationalized. Three real options, then converge.
04

Decide — choose with explicit criteria, not vibes.

Indecision usually isn't a lack of options — it's a lack of stated criteria. Name what you're optimizing for before comparing the options against each other.

💻
Technical Lens

Score each option against explicit criteria: time-to-mitigate, blast radius, reversibility, long-term debt. A reversible decision made fast beats an irreversible one made slow — bias toward reversible.

Document the decision and why — future you (or your team) will re-litigate it without this note.

Chose A (rollback) for immediate mitigation + C (CI gate) for prevention. B deferred — not urgent once A is live.
💗
Personal Lens

Score each option against what actually matters to you: alignment with your values, relationship cost, energy cost, regret-minimization. Ask: "Will I regret not doing this in 5 years?" more than "Is this comfortable right now?"

A decision you can live with and adjust later beats a perfect decision you never make.

Chose C (written boundary first) — respects the relationship, lowers my anxiety about saying it live, still direct.
Shared principle: Circle of Control — spend the decision energy only on what you can actually influence. Everything else is data, not a decision.
05

Do — execute small, verify fast.

The gap between deciding and doing is where most plans die. Shrink the first action until it's genuinely easy to start — then verify before scaling.

💻
Technical Lens

Ship the smallest change that tests the hypothesis. Deploy behind a flag, to a canary, to 1% of traffic — verify the metric moves the way you predicted before rolling out fully.

Write the rollback plan before you deploy, not after something breaks.

Roll back #4471 → confirm p95 returns to ~340ms within 5 min → then schedule the index fix for the next deploy window.
💗
Personal Lens

Take the smallest version of the action today, not the whole plan. A 2-line message beats a perfect unsent essay. An implementation intention — "when X happens, I will do Y" — makes the moment easier to act on.

Decide in advance what you'll do if it goes badly — this removes the fear that's usually the actual blocker.

"When the topic comes up on the next call, I will say: 'I'd rather talk about something else today' — and redirect, calmly, once."
Shared principle: Motion beats meditation. A small real step teaches you more than another hour of planning the perfect one.
06

Debrief — close the loop, or the lesson is lost.

The step everyone skips. Without a debrief, you solved this problem once — and will solve the same class of problem again, from scratch, next time.

💻
Technical Lens

Write a blameless post-mortem: what happened, what the trigger was, what worked, what the structural fix is (Stage 3, option C). Add a regression test so this exact failure can't silently recur.

Update the runbook. The next person — or you, in 6 months — inherits the diagnosis for free.

Post-mortem filed. Load-testing CI gate shipped. Alert threshold lowered from 3s to 1.5s p95 to catch this class earlier next time.
💗
Personal Lens

Reflect without judgment: what actually happened when you took the action? Was the fear accurate? What will you do differently next time this pattern shows up? Write it down — memory reshapes itself without notes.

Notice what worked and name it as a skill you now have, not a lucky one-off.

Said it calmly, redirected once — it worked, no conflict. Fear was worse than reality. Next time: say it sooner, before resentment builds.
Shared principle: The loop only closes if the lesson is written down. An unrecorded lesson is a lesson you'll have to re-learn, at full price, later.
Tools for the toolbox

Eight mental models that apply across both lenses.

Each tagged for where it's most useful — but every one of these works on code and on people.

Both

First Principles

Break the problem down to fundamental truths and reason up from there, instead of reasoning by analogy to how it's "usually done."

Both

The 5 Whys

Ask "why" five times against any symptom. The answer usually stops being obvious around the third one — that's where the real cause lives.

Technical

Rubber Duck Debugging

Explain the problem out loud, line by line, to something that can't respond. Roughly a third of bugs are found before you finish the explanation.

Both

Inversion

Instead of asking "how do I succeed," ask "what would guarantee failure" — then make sure you're not doing any of those things.

Technical

OODA Loop

Observe → Orient → Decide → Act, on a tight cycle. Built for high-uncertainty environments where the situation changes faster than your plan.

Personal

Circle of Control

Sort every worry into: things you control, things you influence, things you don't. Spend energy only on the first two circles.

Both

Pre-Mortem

Before starting, imagine it's six months later and it failed badly. Write down why. Fix the most likely reasons before they happen, not after.

Both

Occam's Razor

Among competing explanations that fit the evidence equally well, the simplest one is usually correct. Suspect the exotic answer before you've ruled out the boring one.

Before you start

Triage first — not every problem needs the full six stages.

Two axes decide how much process a problem deserves: how urgent, and how reversible the decision is.

1
Low urgency + reversible
Just do it. A wrong choice costs almost nothing to undo — save the process for problems that deserve it.
2
High urgency + reversible
Mitigate immediately (Stage 5), then go back and do Stages 1–4 properly for the permanent fix, and debrief.
3
Low urgency + irreversible
This is where the full six-stage framework earns its cost — you have time, and the decision matters. Don't rush it.
4
High urgency + irreversible
The most dangerous quadrant. Slow down enough to get one outside perspective before committing — panic and permanence don't mix.
Where people get stuck

Eight ways this framework quietly breaks.

💻 Technical pitfalls
🩹
Patching the symptom
Restarting the service instead of finding the leak. It "works" until it doesn't, at 3 AM, in production.
🎯
Anchoring on the first hypothesis
Confirmation bias makes you look for evidence that supports your first guess and dismiss evidence that doesn't.
🏗️
Over-engineering the fix
Building a general-purpose framework to solve a problem that happened once. Match the fix to the actual frequency.
📭
Skipping the post-mortem
No written record means the next engineer re-diagnoses the exact same failure from zero.
💗 Personal pitfalls
😶
Avoidance dressed as patience
"I'll deal with it later" often means "I'm avoiding it," and the cost compounds silently while you wait.
🌀
Rumination instead of diagnosis
Replaying the same thought isn't analysis — it's a loop with no exit. Diagnosis has a defined endpoint; rumination doesn't.
🎭
Solving for comfort, not for the goal
Choosing the option that feels least scary right now, instead of the one you'll be glad you chose in five years.
🔁
Never closing the loop
Without reflection, the same relational pattern repeats with a new name attached each time.

The checkout latency spike
production incident · 40 min resolution

02Diagnose: 5 Whys traced it to a new query path missing a DB index, shipped in #4471.
03Diverge: rollback / add index live / ship CI load-test gate — three options at different time horizons.
04Decide: rollback for immediate mitigation, CI gate for long-term prevention. Documented and shared.
05Do: rolled back in 5 min, verified p95 returned to baseline within the next 5 minutes.
06Debrief: blameless post-mortem filed, regression test added, alert threshold tightened.

The recurring family boundary
personal pattern · 3-week arc
01Define: resentment after every call, specifically around career questions — not the calls themselves.
02Diagnose: 5 Whys revealed it wasn't about family judgment — it was my own unresolved doubt about the choice.
03Diverge: say nothing / state it live / write it first — three options with different courage requirements.
04Decide: written boundary first — respects the relationship, lowers anxiety, still direct and clear.
05Do: sent a short, calm message before the next call. Implementation intention made it easy to actually send.
06Debrief: fear was worse than reality — no conflict resulted. Logged the lesson: say it sooner next time.
