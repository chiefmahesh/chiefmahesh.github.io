01
Precision
Be Specific About the Output You Want
Claude takes you literally. A vague prompt returns a generic answer. Specificity — audience, scope, format, depth — produces targeted, usable output.
◆ Core
✗ Vague
explain networking
✓ Specific
explain NAT Gateway in AWS in 3 bullet points for a developer audience
good-prompt.txt
# What makes this specific:
# Audience + Format + Scope + Depth = great output

Explain NAT Gateway in AWS
in 3 bullet points
for a developer audience
who has never used AWS networking before.
Focus on when to use it vs not.
Runs against Claude Sonnet

💡
Checklist before you type: Who is the audience? What format? How long? What angle? Answer these and your prompt writes itself.
02
Context First
Give Context Upfront
State your role, goal, and constraints in the first line. Claude uses everything you give it — so front-load the details that matter most before asking the question.
◆ Core
context-template.txt
# Structure: WHO → GOAL → CONSTRAINT → REQUEST

I am a Director of Engineering.
I want to reduce AWS costs by 30% this quarter.
I have budget authority but no headcount to add.

Analyse this bill and give me the top 5 actionable
savings — ranked by estimated monthly impact.
Include effort level (Low / Med / High) for each.
🎯
The formula: I am [role]. I want [goal]. I have [constraints]. [Request]. — This one structure accounts for 80% of prompt quality improvements.

03
Format Design
Use Examples to Define Format
Instead of describing the format in words, show Claude exactly what you want. One concrete example is worth a paragraph of instructions — and the output will match it precisely.
◆ Core
format-example.txt
# Show the exact structure you want as a template

Output exactly like this for each item:

Service:  EC2
Cost:     $3,879/mo
Action:   Convert on-demand to Reserved Instances
Saving:   ~$1,200/mo (31%)
Effort:   Low — takes 15 min in console

Give me 5 items. No intro paragraph. No conclusion.
Generating…
📋
Pro tip: Paste 2–3 examples of your own past output and say "generate 5 more in this exact style." Claude will reverse-engineer your format and voice almost perfectly.
04
Reasoning
Break Complex Tasks Into Steps
Ask Claude to think step by step. Numbered stages dramatically improve accuracy on anything involving logic, analysis, or prioritisation — it reasons rather than pattern-matches.
◆ Core
step-by-step.txt
# Numbered steps force sequential, logical reasoning

Review this AWS architecture diagram and:

Step 1: List all potential cost issues you spot
Step 2: Prioritise them by monthly dollar impact (high→low)
Step 3: For the top 3 only, suggest a specific fix
Step 4: Flag any risks in implementing those fixes

Do not skip steps. Show your work at each stage.
Generating…
🧠
Magic phrases: "think step by step", "walk me through your reasoning", "before answering, consider both sides". All trigger slower, more accurate reasoning.
05
Structured Input
Use XML Tags for Long or Complex Input
When pasting data, documents, or code, wrap them in XML tags. This gives Claude a clear, unambiguous boundary between your data and your instructions — and it handles this natively.
◈ Pro
xml-pattern.txt
# Wrap pasted data in named XML tags
# Claude treats tag content as clean, bounded input

<aws_bill>
  EC2: $4,230
  RDS: $1,870
  CloudFront: $890
  S3: $340
  Lambda: $120
  NAT Gateway: $680
  Elastic Load Balancing: $450
</aws_bill>

Analyse the <aws_bill> above and find cost anomalies.
For each, explain why it looks high and suggest a fix.
Order by potential monthly saving, largest first.
Generating…
🏷️
Use descriptive tag names: <contract>, <error_log>, <email_thread>. Multiple tags in one prompt is fine — Claude tracks each one independently.
06
Guardrails
Set Constraints Explicitly
Constraints are as powerful as instructions. Defining word limits, regions, prohibited terms, and forbidden formats sculpts the output before a single word is generated.
◆ Core
constraints.txt
# List constraints as explicit rules — Claude follows them reliably

Summarise our AWS cost anomalies for the board.

Constraints:
- Reply in under 200 words
- No bullet points — plain prose only
- Focus only on Mumbai (ap-south-1) region costs
- Avoid technical jargon — board members are non-technical
- Do not include any cost figures under $500/mo
- End with exactly one recommended immediate action

07
Persona
Assign a Role
Telling Claude to act as a specific expert instantly calibrates vocabulary, depth, and perspective. The same question gets a radically different — and better — response when filtered through a specialist persona.
◆ Core
role-assignment.txt
# Specific roles → specific expertise and tone

Act as a senior AWS cost optimisation consultant
with 10 years at Accenture specialising in fintech.
You are known for blunt, direct advice with no padding.

Review this architecture and give your honest assessment.
Do not soften your critique. If something is wasteful,
say it plainly and explain the financial impact.
Generating…
🎭
Roles that work great: "You are a patient teacher explaining to a 12-year-old", "You are a senior lawyer reviewing this clause", "You are a ruthless editor", "You are a skeptical investor" — each transforms the tone and quality.
08
Momentum
Iterate — Don't Start Over
Claude holds the full conversation in memory. Instead of retyping from scratch, build on what it just produced. Short follow-up instructions are far more efficient than rewriting the whole prompt.
◇ Advanced
iteration-flow.txt
# Turn 1: Initial request
Analyse our AWS bill and list the top 5 cost savings.

# Turn 2: Narrow scope
Good. Now do the same for CloudWatch only.

# Turn 3: Change output
Make it shorter — one sentence per item maximum.

# Turn 4: Get actionable output
Give me the CLI commands to implement fix #2.

# Turn 5: Transform format
Convert this into a Jira ticket description.
Generating…
⚡
Power moves: "Make it half as long", "Now write it for a non-technical audience", "Add a risk column to that table", "Convert to a Slack message" — all work perfectly without re-explaining the context.
09
Negative Space
Use "Negative" Instructions
Telling Claude what NOT to do is often more powerful than describing what to do. Eliminating default behaviours — the summary paragraph, the disclaimer, the question echo — sharpens every response.
◇ Advanced
negatives.txt
# Strip default AI padding with explicit negatives

Review our AWS cost report and give recommendations.

Do not repeat the question back to me.
Do not add a summary or conclusion paragraph at the end.
Do not suggest consulting an AWS Solutions Architect.
Do not hedge with phrases like "it depends" or "it varies".
Do not use em-dashes or parenthetical asides.
Do not list more than 5 items.
Generating…
🚫
Common AI habits to kill: "It's important to note that...", "In conclusion...", "Great question!", "As an AI...". One "do not" wipes each one permanently for that conversation.
10
Optionality
Ask for Alternatives
Request multiple approaches — and label the risk appetite for each. You get a full decision landscape instead of one opinion, and you stay in control of which path to take.
◇ Advanced
alternatives.txt
# Labelled alternatives give you a decision landscape

Give me 3 different approaches to reduce ECR costs:

Option A — Aggressive:  Maximum saving, accepts risk
Option B — Moderate:   Balanced saving vs stability
Option C — Zero-risk:  No downtime, fully reversible

For each option:
- Estimated monthly saving
- Biggest risk or trade-off
- Time to implement
- One CLI command to start
Generating…
🔀
More alternative frames: "fast vs thorough", "cheap vs robust", "now vs later", "for a startup vs enterprise", "assuming 100 users vs 10M users" — each slices the problem space differently.
